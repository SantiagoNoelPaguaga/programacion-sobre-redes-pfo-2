Descripción del Proyecto

Implementación de una API REST funcional para la gestión de tareas. El sistema cuenta con una arquitectura híbrida que garantiza portabilidad local y disponibilidad en la nube.

Local: Flask + Consola
Cloud: VPS Oracle + Docker
Web: GitHub Pages
Seguridad: Nginx + SSL

Tecnologías Utilizadas

Python 3 / Flask
SQLite (Persistencia)
Werkzeug Security (Hashing)
Docker & Docker Compose
Nginx & Certbot (SSL)
GitHub Pages (Frontend)

Cómo ejecutar el proyecto

1. Ejecución Local (Consola)

Clonar e instalar dependencias:

pip install -r requirements.txt

Ejecutar servidor:

python3 servidor.py

Ejecutar cliente:

python3 cliente.py

Pruebas de Funcionamiento

1. Inicialización del Sistema en Consola

Arranque inicial de los componentes del entorno local. En la terminal izquierda se ejecuta servidor.py y en la derecha el cliente.py, desplegando el menú de interacción para el usuario.

2. Validación de Integridad (HTTP 400)

Demostración de la lógica de validación en el endpoint POST/registro. Al intentar registrar un usuario ya existente en la base de datos SQLite, el servidor responde con un error 400 (Bad Request).

3. Registro Exitoso de Usuario (HTTP 201)

Proceso de alta exitosa para un nuevo usuario. El servidor confirma la creación del recurso con un código 201 y almacena los datos de forma persistente.

4. Protección de Rutas (HTTP 401)

Verificación de seguridad en el endpoint GET/tareas. El sistema bloquea el acceso con un código 401 si no detecta una sesión activa, garantizando que el contenido sea privado.

5. Intento de Inicio de Sesión Fallido

Control de autenticación. Al ingresar credenciales incorrectas en el endpoint POST/login, el servidor deniega el acceso para proteger la cuenta del usuario.

6. Inicio de Sesión Exitoso (HTTP 200)

Autenticación exitosa mediante la verificación de credenciales. El servidor responde con un código 200 y activa la sesión para el usuario autenticado.

7. Acceso Autorizado y Respuesta HTML

Visualización del recurso protegido. El servidor entrega un documento HTML de bienvenida personalizado tras validar satisfactoriamente la sesión del usuario.

8. Persistencia y Seguridad (DB Browser)

Evidencia del almacenamiento en la base de datos database.db. Se observa que las contraseñas están protegidas mediante hashing, cumpliendo con los estándares de seguridad requeridos.

Respuestas Conceptuales

¿Por qué hashear contraseñas?

Es una función unidireccional que protege la información sensible. Si la base de datos fuera vulnerada, el atacante solo encontraría cadenas ilegibles.

Ventajas de usar SQLite

Persistencia directa en un solo archivo, sin configuración de motores externos y alta portabilidad.
